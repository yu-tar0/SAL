# 構造化データ運用

> One-line summary: raw と wiki の間に DuckDB による構造化データ層を置き、比較・集計できる知識を管理する運用ルール。

## TL;DR

`raw/` は一次資料、`wiki/` は要約と解釈、`db/` は比較・集計可能な構造化データを担う。最初は `db/knowledge.duckdb` 1つを標準 DB とし、スキーマは `db/schema.sql` を正とする。ドメイン固有の表は最初から作り込まず、`sources`、`entities`、`observations`、`relationships` を中心に始める。

保存と人間編集は `raw/data/` の CSV / Excel / JSON / Parquet、活用と分析は `db/knowledge.duckdb` を基本とする。定量仕様は DuckDB、短い定性メモは DuckDB の `TEXT` や `notes`、長文メモや思考過程は `wiki/` の Markdown に置き、DB 側の `wiki_path` や `note_id` から相互参照する。原本の具体位置へは `raw_path` と `raw_locator` で辿る。

## 概要

Markdown wiki は文脈や解釈を残すのに向いているが、数値比較、条件抽出、ランキング、時系列集計には弱い。そのため、定量情報や関係性を DuckDB に置き、分析結果や解釈を wiki に戻す。

## 主要 claim

- `raw/` は原本保護を担い、既存ファイルは編集しない。
- `db/` は構造化された観察事実を保持し、SQL で比較・集計できるようにする。
- `wiki/` は DB の値を説明する文脈、要約、定性情報、未解決の問いを保持する。
- DB の各行は、可能な限り `sources` または `raw/` 上の原本へ辿れるようにする。
- 行単位で原本の具体位置を示す必要がある場合は、`raw_path` と `raw_locator` を使う。
- 長文・自由記述・思考メモは DB に閉じ込めず、Markdown に置いたうえで `wiki_path` や `note_id` から参照する。
- 矛盾する値は上書きせず、`conflict_status` や wiki 本文で `競合` / `未確定` として残す。

## 詳細

### 標準構成

- `raw/data/`: CSV、Excel、JSON、Parquet などの元データ。
- `raw/archives/`: 古い元データ、廃止版、再現のために残す旧版。
- `db/schema.sql`: DuckDB の標準スキーマ。
- `db/knowledge.duckdb`: 標準 DB。必要になった時点で生成する。

### 最初のテーブル

- `sources`: 原本や取得元のメタデータ。
- `imports`: raw データを DB に取り込んだ履歴。
- `entities`: 人物、組織、製品、論文、データセット、概念などの固有項目。
- `entity_aliases`: 表記ゆれや別名。
- `observations`: 数値、テキスト値、日付、分類などの観察事実。
- `relationships`: entity 間の関係。

### Wiki との接続

wiki ページには、必要に応じて frontmatter に `db_entity_id` や `db_source_id` を置く。DB の値を本文に転記する場合は、どの source / raw に基づく値かを残す。

DB 側から wiki へ辿るときは、代表ページには `entities.wiki_path`、行単位の長文メモには `observations.wiki_path` / `relationships.wiki_path` と `note_id` を使う。`note_id` は Markdown 内の見出し、frontmatter、または本文中の明示 ID と対応させ、同じメモを後から再特定できるようにする。

DB 側から raw へ辿るときは、通常は `source_id` から `sources.source_path` を参照する。観察事実や関係性が raw の特定ページ、画像、CSV 行、動画時刻などに紐づく場合は、行側に `raw_path` と `raw_locator` を持たせる。`raw_path` は原本ファイル、`raw_locator` はその中の位置を表し、例として `page=12`、`row=45`、`timestamp=00:12:34`、`section=3.2` などを使う。

`raw_path` / `raw_locator` は原本の場所を示し、`wiki_path` / `note_id` は解釈・長文メモの場所を示す。両者は役割が異なるため、同じ行に併存してよい。

### 定性メモの置き場所

| 種類 | 主な置き場所 | 補足 |
|---|---|---|
| 定量仕様、日付、分類、状態 | DuckDB の固定列または `observations` | 比較、集計、抽出に使う。 |
| 短い定性メモ | DuckDB の `notes` や `value_text` | 一覧表示や SQL 検索で使う補足に限る。 |
| 中程度の構造化メモ | DuckDB の `TEXT`、必要に応じて JSON 相当の文字列 | 頻繁に検索・集計する項目は固定列化を検討する。 |
| 長文、自由記述、判断理由、思考メモ | `wiki/` の Markdown | DB 側には `wiki_path` と `note_id` を残し、本文は wiki に置く。 |
| 原本中の該当箇所 | `raw_path` と `raw_locator` | 原本ファイルとページ、行、時刻などの位置を示す。 |

### 競合・未確定

同じ entity と attribute に異なる値が出た場合は、古い値を黙って消さない。DB では `conflict_status` を使い、wiki では `競合` または `未確定` として人間確認事項にする。
